Recommended patterns

In order to make the best out of formo we recommend a few patterns.

Avoid using native form elements directly#

formo's utility functions are designed to return precise and type-safe data. For instance, you may notice that onChange accepts a precise type (matching value), as opposed to a more generic Event.

For this reason, we recommend defining your own wrappers for native form elements, in order to gain type-safety.

It's likely that you are already be doing this in order to provide other customizations for your project, so this is normally not a great change.

formo provides a FieldProps type that helps you create custom components for your form elements.

For example, you may define your own TextField field as:

tsx
import { FieldProps, NonEmptyArray } from "@buildo/formo";
Â 
type Props = FieldProps<string, string, NonEmptyArray<string>> & {
  placeholder: string;
};
Â 
export function TextField(props: Props) {
  return (
    <div>
      <label>{props.label}</label>
      <input
        name={props.name}
        type="text"
        value={props.value}
        onChange={(e) => props.onChange(e.currentTarget.value)}
        onBlur={props.onBlur}
        disabled={props.disabled}
      />
      <ul>
        {props.issues?.map((issue) => (
          <li key={`${props.name}_${issue}`}>{issue}</li>
        ))}
      </ul>
    </div>
  );
}

tip

You'll notice that FieldProps has three type parameters.

  • The first one is the value type.
  • The second one is the label type (which may or may not be a string)
  • The third one is the field error type (wrapped in an NonEmptyArray)

In this example we've used string for all of them, but in a real project you may have more specific types for the label (think of something like LocalizedString, a string which has been localized) and for the field error (perhaps a complex error object with additional info)

Once you've done that, you can now see how formo utility methods (such as fieldProps) become very convenient to use:

tsx
import { useFormo, validators, success } from "@buildo/formo";
Â 
export const MyForm = () => {
  const { fieldProps } = useFormo(
    {
      initialValues: {
        name: "",
      },
      fieldValidators: () => ({
        name: validators.maxLength(10, "Name is too long"),
      }),
    },
    {
      onSubmit: async (values) => success(values),
    }
  );
Â 
  return (
    <form>
      <TextField
        label="Name"
        placeholder="Ada Lovelace"
        {...fieldProps("name")}
      />
    </form>
  );
};

Define a shared type for field issues#

formo is agnostic in the specific type you use for field issues, but it's highly recommended to define a fixed type for this and re-use it across your application.

For instance:

ts
export type Severity = "error" | "warning";
export type FieldIssue = { message: string; severity: Severity };
Â 
export const error = (message: string): FieldIssue => ({
  message,
  severity: "error",
});
Â 
export const warning = (message: string): FieldIssue => ({
  message,
  severity: "warning",
});

ts
import { useFormo, validators, success } from "@buildo/formo";

    }),
  },
  {
    onSubmit: async (values) => success(values),
  }
);
